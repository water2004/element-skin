# 权限模型设计方案

## 1. 设计目标

权限系统应成为站点所有业务操作的统一安全边界。每个用户拥有一个权限主体，每次需要授权的业务操作都携带权限主体，并由业务层根据权限主体和本次会话入口共同判断是否允许执行。

本方案只定义权限模型本身，不包含具体实现排期。

目标：

- 每个用户必须拥有一个权限主体。
- 每个需要授权的操作都必须携带权限主体。
- 权限判断不能散落在路由层、数据库层或临时布尔字段中。
- 管理员能力必须细粒度化，不能只靠“是否管理员”判断。
- 超级管理员不绕过权限检查，只是通过角色拥有完整权限集合。
- Yggdrasil、Microsoft 登录、站内网页登录都必须落到同一套权限主体模型。
- 权限数据必须结构化存储，不能用 JSON 字符串规避建模。
- 旧的 `is_admin`、`is_super_admin` 风格接口不作为长期兼容层保留。

## 2. 核心原则

### 2.1 权限主体是用户能力的唯一来源

权限主体代表“这个用户在站内最终具备哪些能力”。业务层不应继续接收裸 `user_id`、`is_admin`、`is_super_admin` 后自行拼权限逻辑。

允许：

```go
func UpdateProfile(ctx context.Context, actor PermissionActor, profileID string, input UpdateProfileInput) error
```

不允许：

```go
func UpdateProfile(ctx context.Context, userID string, isAdmin bool, profileID string, input UpdateProfileInput) error
```

### 2.2 会话入口限制本次请求能使用哪些权限

权限主体说明用户拥有什么能力，会话入口说明本次请求能使用哪些能力。

例如同一个用户：

- 通过网页登录时，可以使用完整站内用户权限。
- 通过 Yggdrasil access token 调用时，只能使用与该令牌、绑定角色和 Yggdrasil 协议相关的权限。
- 通过 Microsoft 登录回调产生的临时状态时，只能完成绑定或导入流程，不能变成完整站内会话。

最终可用权限：

```text
actor_permissions = subject_effective_permissions ∩ session_allowed_permissions ∩ session_resource_constraints
```

这可以避免“用户有权限，所以任意 token 都能做所有事”的问题。

### 2.3 权限检查不能被身份绕开

超级管理员、普通管理员和普通用户都走同一套权限检查。

超级管理员之所以能做所有事，是因为它的权限主体拥有所有权限，而不是因为代码中存在：

```go
if actor.IsSuperAdmin {
    return nil
}
```

这种短路会让权限模型失真，也会让审计和测试失去统一边界。

### 2.4 角色只是权限模板

角色用于批量授予权限，但业务层最终只判断有效权限集合。

业务层不应写：

```go
if actor.HasRole("admin") {
    ...
}
```

业务层应写：

```go
Require(actor, "notice.create.any")
```

### 2.5 权限范围必须一开始建模清楚

权限范围不能只做 `self` 和 `any`。这两个范围不足以表达：

- Yggdrasil token 只能操作绑定角色。
- 用户只能管理自己拥有的材质。
- 公开皮肤库可被任意登录用户读取。
- 内容审核员只能处理被分配或公开队列中的资源。
- 后台任务只能执行系统维护能力。

因此权限范围应作为独立注册表建模，而不是写死为两个字符串。

## 3. 权限 Actor 模型

### 3.1 PermissionActor

业务层接收的不是裸权限主体，而是 PermissionActor。

PermissionActor = 权限主体 + 会话入口 + 本次请求约束。

建议字段：

```text
subject_id
user_id
session_kind
session_id
entrypoint
bound_profile_id
permissions
constraints
```

说明：

- `subject_id`：权限主体 ID。
- `user_id`：主体归属用户。
- `session_kind`：本次请求来源，例如 `web`、`yggdrasil`、`microsoft_state`、`system`。
- `session_id`：站内会话、Yggdrasil token、临时状态或系统任务 ID。
- `entrypoint`：更细的入口标识，例如 `dashboard`、`admin_panel`、`authserver`、`sessionserver`、`microsoft_import`。
- `bound_profile_id`：Yggdrasil token 绑定的角色，可为空。
- `permissions`：已经计算并受会话入口约束后的权限集合。
- `constraints`：本次请求的结构化约束，不使用 JSON 字符串。

### 3.2 PermissionSubject

PermissionSubject 是用户的长期权限身份。

建议字段：

```text
id
user_id
status
created_at
updated_at
```

`status` 可取：

```text
active
disabled
locked
```

说明：

- `active`：主体正常可用。
- `disabled`：主体被禁用，所有需要认证的操作应拒绝。
- `locked`：主体暂时锁定，可用于安全风控或管理员处理。

### 3.3 权限主体和用户的关系

用户表只保留账号身份信息。权限不应继续散落在用户表中。

用户创建时必须同时创建权限主体，并授予默认角色。

```text
users
  -> permission_subjects
      -> subject_roles
      -> subject_permission_overrides
```

### 3.4 会话入口和主体的关系

不同入口都必须归一到 PermissionActor。

```text
站内网页登录
  -> user_id
  -> permission_subject
  -> session_kind=web
  -> PermissionActor

Yggdrasil access token
  -> token
  -> user_id + bound_profile_id
  -> permission_subject
  -> session_kind=yggdrasil
  -> PermissionActor

Microsoft 临时导入状态
  -> state token
  -> user_id + external_profile_claim
  -> permission_subject
  -> session_kind=microsoft_state
  -> PermissionActor
```

## 4. Permission 模型

### 4.1 Permission 命名

Permission 是系统中可判断的最小授权单元。

命名格式：

```text
resource.action.scope
```

示例：

```text
profile.read.owned
profile.update.owned
profile.update.bound_profile
texture.read.public
texture.create.owned
texture.apply.bound_profile
notice.read.owned
notice.create.any
site.settings.update.any
yggdrasil.session.create.owned
yggdrasil.server.join.bound_profile
microsoft.profile.import.owned
```

### 4.2 resource

资源表示权限作用的业务对象。

建议首批资源：

```text
account
profile
texture
wardrobe
notice
user
site.settings
permission
audit
yggdrasil.session
yggdrasil.server
microsoft.identity
microsoft.profile
```

### 4.3 action

动作表示可执行行为。

常用动作：

```text
read
create
update
delete
grant
revoke
review
publish
archive
bind
unbind
import
join
validate
refresh
invalidate
apply
clear
```

动作应贴合业务语义，不要为了复用而过度抽象。

### 4.4 scope

scope 表示权限作用范围。它必须独立建模，不能只靠代码里的字符串约定。

建议首批 scope：

```text
self
owned
bound_profile
public
assigned
any
system
```

说明：

- `self`：账号自身，例如修改自己的邮箱、密码、偏好设置。
- `owned`：归属于当前用户的资源，例如自己的角色、材质、衣柜。
- `bound_profile`：被当前会话明确绑定的角色，只适用于带角色绑定的入口，例如 Yggdrasil token。
- `public`：公开资源，例如公开皮肤库、公开角色查询。
- `assigned`：被分配给当前主体处理的资源，例如审核任务。
- `any`：任意用户或全站资源，通常只授予管理类角色。
- `system`：仅系统任务主体可用，例如过期数据清理。

scope 不是随意字符串，必须在 `permission_scopes` 中登记，并由明确的 resolver 解释。

## 5. Scope Resolver

### 5.1 为什么需要 resolver

`owned`、`bound_profile`、`assigned` 这类范围不能只通过权限 ID 判断。它们必须结合资源数据和本次会话上下文。

例如：

```text
profile.update.owned
```

需要读取 profile 的 `user_id`，确认它等于 actor 的 `user_id`。

```text
profile.update.bound_profile
```

需要确认 profile 的 `id` 等于 actor 的 `bound_profile_id`。

### 5.2 resolver_key

每个 scope 需要一个 resolver。

```text
self          -> subject_user
owned         -> resource_owner
bound_profile -> session_bound_profile
public        -> resource_public
assigned      -> resource_assignment
any           -> global
system        -> system_actor
```

Resolver 是代码能力，但 resolver 与 scope 的绑定是结构化数据。

## 6. 默认角色模型

### 6.1 Role

Role 是权限集合模板。

建议字段：

```text
id
name
description
system_role
created_at
updated_at
```

`system_role` 表示系统内置角色。内置角色可以调整权限绑定，但不应被删除。

### 6.2 默认角色

建议内置角色：

```text
user
developer
moderator
admin
super_admin
system_maintenance
```

### 6.3 user

普通用户默认能力：

```text
account.read.self
account.update.self
profile.read.owned
profile.create.owned
profile.update.owned
profile.delete.owned
texture.read.owned
texture.read.public
texture.create.owned
texture.update.owned
texture.delete.owned
wardrobe.read.owned
wardrobe.update.owned
notice.read.owned
yggdrasil.session.create.owned
microsoft.identity.bind.self
microsoft.identity.unbind.self
microsoft.profile.import.owned
```

### 6.4 yggdrasil_session

Yggdrasil access token 不应直接使用完整 user 角色权限。它应通过会话入口限制成协议专用权限。

建议入口允许权限：

```text
yggdrasil.session.refresh.owned
yggdrasil.session.validate.owned
yggdrasil.session.invalidate.owned
yggdrasil.server.join.bound_profile
profile.read.bound_profile
texture.apply.bound_profile
texture.clear.bound_profile
```

这些权限不是单独授予用户的长期角色，而是 Yggdrasil 会话入口对 actor 权限的进一步收窄。

### 6.5 moderator

内容审核角色：

```text
texture.read.public
texture.update.assigned
texture.delete.assigned
texture.review.assigned
```

如需要全站内容处理，再授予：

```text
texture.read.any
texture.update.any
texture.delete.any
```

### 6.6 admin

管理员角色：

```text
user.read.any
user.update.any
profile.read.any
profile.update.any
texture.read.any
texture.update.any
texture.delete.any
notice.create.any
notice.update.any
notice.delete.any
site.settings.read.any
audit.read.any
permission.read.any
```

### 6.7 super_admin

超级管理员角色拥有系统内所有 Permission。

注意：

- 代码不允许通过 `is_super_admin` 短路权限判断。
- 超级管理员权限来自角色权限绑定。
- 新增 Permission 时必须同步加入 `super_admin` 角色。
- 转移或变更超级管理员本身应是受控业务流程，不能通过普通角色编辑绕过。

### 6.8 system_maintenance

系统维护主体用于计划任务，不归属于普通用户。

建议权限：

```text
notice.delete.system
session.delete.system
cache.invalidate.system
audit.archive.system
```

系统维护主体也必须走权限模型，不能在任务代码里绕过权限系统。

## 7. Yggdrasil 纳入方式

Yggdrasil 不是站外授权模型，而是站内用户的游戏客户端会话入口。

### 7.1 authenticate

认证流程：

```text
用户名/邮箱/角色名 + 密码
  -> 找到 user
  -> 找到 permission_subject
  -> 检查 subject 是否 active
  -> 检查 yggdrasil.session.create.owned
  -> 创建 Yggdrasil token
```

如果用户没有 `yggdrasil.session.create.owned`，即使密码正确，也不能创建游戏客户端会话。

### 7.2 selectedProfile

如果 token 绑定了角色：

```text
token.user_id = user.id
token.bound_profile_id = profile.id
```

后续使用该 token 时，PermissionActor 必须包含：

```text
session_kind = yggdrasil
bound_profile_id = profile.id
```

### 7.3 refresh / validate / invalidate

这些操作只需要验证 token 所属用户的权限主体仍然 active，并且 actor 具备：

```text
yggdrasil.session.refresh.owned
yggdrasil.session.validate.owned
yggdrasil.session.invalidate.owned
```

如果用户被禁用、权限被移除或主体 locked，应拒绝刷新和继续使用 token。

### 7.4 join

进入服务器必须使用绑定角色权限：

```text
yggdrasil.server.join.bound_profile
```

校验：

- token 有效。
- token 绑定了 `bound_profile_id`。
- 请求中的 selectedProfile 等于 `bound_profile_id`。
- actor 具备 `yggdrasil.server.join.bound_profile`。

### 7.5 材质设置

通过 Yggdrasil API 修改皮肤或披风时，不应使用普通网页登录权限，而应使用：

```text
texture.apply.bound_profile
texture.clear.bound_profile
```

Resolver 必须确认：

- profile.id 等于 actor.bound_profile_id。
- profile.user_id 等于 actor.user_id。

## 8. Microsoft 登录纳入方式

Microsoft 登录和导入是站内用户证明外部身份的流程，不应产生新的权限主体。

### 8.1 auth-url

用户请求 Microsoft 登录地址时，需要站内 web actor，并检查：

```text
microsoft.identity.bind.self
```

返回的 state 必须绑定：

```text
user_id
subject_id
purpose
created_at
expires_at
```

### 8.2 callback

Microsoft callback 是临时认证结果入口。它不等于站内完整登录，也不能绕过权限模型。

callback 成功后产生一个短期 `microsoft_state`，它只能用于后续查询或导入 Microsoft profile。

### 8.3 get-profile

获取 Microsoft profile 时，必须同时满足：

- 当前站内 web actor 有效。
- `microsoft_state.user_id == actor.user_id`。
- actor 具备 `microsoft.profile.import.owned` 或专门的读取权限。

### 8.4 import-profile

导入 Microsoft 角色时，检查：

```text
microsoft.profile.import.owned
profile.create.owned
```

导入后创建的 profile 归属于 actor.user_id。

Microsoft state 只能消费一次。失败时不能污染 profile、texture 或绑定状态。

## 9. 权限判断模型

### 9.1 Require

用于判断某 actor 是否具备具体权限。

```text
Require(actor, permission)
```

规则：

- subject 状态不是 `active`，拒绝。
- actor permissions 包含 permission，允许。
- 否则拒绝。

### 9.2 RequireScoped

用于需要资源范围解析的权限。

```text
RequireScoped(actor, permission, resource)
```

规则：

- actor permissions 包含 permission。
- permission.scope 对应 resolver 通过。
- 否则拒绝。

示例：

```text
RequireScoped(actor, "profile.update.owned", profile)
RequireScoped(actor, "profile.update.bound_profile", profile)
RequireScoped(actor, "texture.update.assigned", reviewTask)
```

### 9.3 RequireAnyScoped

用于多个权限满足一个即可的资源操作。

例如管理员或资源拥有者都能更新角色：

```text
RequireAnyScoped(actor, [
  "profile.update.owned",
  "profile.update.any"
], profile)
```

其中：

- `owned` 需要 resolver 校验归属。
- `any` 使用 global resolver。

## 10. 数据库模型

### 10.1 permission_subjects

```sql
CREATE TABLE permission_subjects (
    id TEXT PRIMARY KEY,
    user_id TEXT UNIQUE,
    kind TEXT NOT NULL CHECK(kind IN ('user', 'system')),
    status TEXT NOT NULL CHECK(status IN ('active', 'disabled', 'locked')),
    created_at BIGINT NOT NULL,
    updated_at BIGINT NOT NULL,
    FOREIGN KEY(user_id) REFERENCES users(id) ON DELETE CASCADE
);
```

`kind='system'` 时 `user_id` 为空，用于计划任务等系统主体。

### 10.2 permission_scopes

```sql
CREATE TABLE permission_scopes (
    id TEXT PRIMARY KEY,
    description TEXT NOT NULL DEFAULT '',
    resolver_key TEXT NOT NULL,
    created_at BIGINT NOT NULL
);
```

首批数据：

```text
self          subject_user
owned         resource_owner
bound_profile session_bound_profile
public        resource_public
assigned      resource_assignment
any           global
system        system_actor
```

### 10.3 permissions

```sql
CREATE TABLE permissions (
    id TEXT PRIMARY KEY,
    resource TEXT NOT NULL,
    action TEXT NOT NULL,
    scope_id TEXT NOT NULL,
    description TEXT NOT NULL DEFAULT '',
    created_at BIGINT NOT NULL,
    UNIQUE(resource, action, scope_id),
    FOREIGN KEY(scope_id) REFERENCES permission_scopes(id) ON DELETE RESTRICT
);
```

### 10.4 roles

```sql
CREATE TABLE roles (
    id TEXT PRIMARY KEY,
    name TEXT NOT NULL UNIQUE,
    description TEXT NOT NULL DEFAULT '',
    system_role BOOLEAN NOT NULL DEFAULT FALSE,
    created_at BIGINT NOT NULL,
    updated_at BIGINT NOT NULL
);
```

### 10.5 role_permissions

```sql
CREATE TABLE role_permissions (
    role_id TEXT NOT NULL,
    permission_id TEXT NOT NULL,
    created_at BIGINT NOT NULL,
    PRIMARY KEY(role_id, permission_id),
    FOREIGN KEY(role_id) REFERENCES roles(id) ON DELETE CASCADE,
    FOREIGN KEY(permission_id) REFERENCES permissions(id) ON DELETE CASCADE
);
```

### 10.6 subject_roles

```sql
CREATE TABLE subject_roles (
    subject_id TEXT NOT NULL,
    role_id TEXT NOT NULL,
    granted_by_subject_id TEXT,
    created_at BIGINT NOT NULL,
    PRIMARY KEY(subject_id, role_id),
    FOREIGN KEY(subject_id) REFERENCES permission_subjects(id) ON DELETE CASCADE,
    FOREIGN KEY(role_id) REFERENCES roles(id) ON DELETE CASCADE,
    FOREIGN KEY(granted_by_subject_id) REFERENCES permission_subjects(id) ON DELETE SET NULL
);
```

### 10.7 subject_permission_overrides

```sql
CREATE TABLE subject_permission_overrides (
    subject_id TEXT NOT NULL,
    permission_id TEXT NOT NULL,
    effect TEXT NOT NULL CHECK(effect IN ('allow', 'deny')),
    granted_by_subject_id TEXT,
    created_at BIGINT NOT NULL,
    PRIMARY KEY(subject_id, permission_id),
    FOREIGN KEY(subject_id) REFERENCES permission_subjects(id) ON DELETE CASCADE,
    FOREIGN KEY(permission_id) REFERENCES permissions(id) ON DELETE CASCADE,
    FOREIGN KEY(granted_by_subject_id) REFERENCES permission_subjects(id) ON DELETE SET NULL
);
```

### 10.8 session_permission_policies

用于描述不同会话入口允许使用哪些权限。

```sql
CREATE TABLE session_permission_policies (
    session_kind TEXT NOT NULL,
    permission_id TEXT NOT NULL,
    created_at BIGINT NOT NULL,
    PRIMARY KEY(session_kind, permission_id),
    FOREIGN KEY(permission_id) REFERENCES permissions(id) ON DELETE CASCADE
);
```

例如 `session_kind='yggdrasil'` 只允许 Yggdrasil 相关权限和 `bound_profile` 范围权限。

### 10.9 permission_audit_logs

权限变更必须可审计。

```sql
CREATE TABLE permission_audit_logs (
    id TEXT PRIMARY KEY,
    actor_subject_id TEXT,
    action TEXT NOT NULL,
    target_subject_id TEXT,
    target_role_id TEXT,
    target_permission_id TEXT,
    created_at BIGINT NOT NULL,
    FOREIGN KEY(actor_subject_id) REFERENCES permission_subjects(id) ON DELETE SET NULL,
    FOREIGN KEY(target_subject_id) REFERENCES permission_subjects(id) ON DELETE SET NULL,
    FOREIGN KEY(target_role_id) REFERENCES roles(id) ON DELETE SET NULL,
    FOREIGN KEY(target_permission_id) REFERENCES permissions(id) ON DELETE SET NULL
);
```

第一阶段审计表只记录结构化目标。不要用 JSON 字符串塞任意详情。

## 11. 权限计算

### 11.1 Subject Effective Permissions

主体长期有效权限：

```text
role_allow_permissions
+ subject_allow_permissions
- subject_deny_permissions
```

计算步骤：

1. 读取主体状态。
2. 主体不是 `active` 则返回空权限集合。
3. 读取主体角色。
4. 读取角色绑定权限。
5. 读取主体权限覆盖项。
6. 合并 allow。
7. 移除 deny。
8. 返回 subject effective permissions。

### 11.2 Actor Permissions

本次请求可用权限：

```text
subject_effective_permissions
∩ session_permission_policy(session_kind)
```

站内 web session 的 policy 可以包含完整用户可用权限。  
Yggdrasil session 的 policy 只能包含协议相关能力。  
Microsoft state 的 policy 只能包含绑定、查询、导入相关能力。

超级管理员不特殊短路。  
`super_admin` 角色应绑定所有权限，因此 subject effective permissions 自然包含所有权限；但如果它通过 Yggdrasil token 访问，仍会被 session policy 收窄。

## 12. 权限缓存

权限集合可以缓存，但数据库是唯一事实来源。

建议缓存：

```text
permission:subject:{subject_id}:effective
permission:actor:{session_kind}:{session_id}
permission:user:{user_id}:subject
```

必须失效的场景：

- 主体状态变化。
- 主体角色变化。
- 角色权限变化。
- 主体权限覆盖项变化。
- 新增或删除权限。
- scope resolver 绑定变化。
- session policy 变化。
- 用户密码修改、用户禁用、用户删除。
- Yggdrasil token 刷新、失效、绑定角色变化。
- Microsoft state 被消费或过期。

缓存失效是权限变更流程的一部分，不能作为可选优化。

## 13. 业务层使用规范

### 13.1 Service 必须接收 PermissionActor

涉及用户数据、站点配置、管理行为、公开内容变更的 service 方法必须接收 `PermissionActor`。

只读公开接口可以不接收 actor，但必须明确标记为公开能力。

### 13.2 权限判断必须靠近业务资源

资源归属需要读数据库后才能确定，因此权限判断通常应在 service 层：

```text
读取 profile
  -> 不存在返回 404
  -> RequireAnyScoped(profile.update.owned, profile.update.any, profile)
  -> 执行更新
```

### 13.3 路由层不做业务权限

路由层只负责：

- 解析请求。
- 认证会话入口。
- 构造 PermissionActor。
- 调用 service。
- 转换响应。

路由层不应直接判断角色或权限，除非是非常薄的认证包装。

## 14. 管理页面模型

权限系统需要支持以下管理能力：

- 查看所有角色。
- 查看角色包含的权限。
- 查看所有 scope 和 resolver 说明。
- 修改非系统角色。
- 查看用户权限主体。
- 给用户授予或移除角色。
- 给用户添加 allow/deny 权限覆盖项。
- 查看 session policy。
- 查看权限审计日志。

超级管理员相关管理规则：

- 超级管理员角色不能被普通管理员编辑。
- 超级管理员主体不能被普通管理员禁用。
- 超级管理员转移应使用独立业务流程。

## 15. 测试要求

权限模型的测试必须验证精确结果。

最低测试矩阵：

- 用户创建时创建权限主体。
- 默认用户拥有普通用户角色。
- 普通用户能操作 owned 资源。
- 普通用户不能操作他人的 owned 资源。
- 拥有 `any` 权限的主体能操作他人资源。
- `public` 资源可以按 public resolver 读取。
- `assigned` 资源必须有明确分配关系。
- `bound_profile` 必须匹配 actor.bound_profile_id。
- 显式 deny 覆盖角色 allow。
- 主体 disabled 后所有权限失效。
- 角色增加权限后主体有效权限变化。
- 角色移除权限后主体有效权限变化。
- 超级管理员通过角色拥有所有权限，而不是通过短路。
- 超级管理员通过 Yggdrasil token 访问时仍被 session policy 收窄。
- Yggdrasil token 不能操作未绑定角色。
- Yggdrasil token 在主体 locked 后不能 refresh 或 join。
- Microsoft state 只能给同一 user_id 使用。
- Microsoft state 消费后不能重放。
- 权限变更后缓存失效。
- 权限不足时数据库不发生写入。

不允许只验证“状态码不是 500”或“结果非空”。

## 16. 命名规范

Permission ID 使用小写英文、点号分隔：

```text
resource.action.scope
```

禁止：

```text
CanManageUser
admin
super
user_write
```

角色 ID 使用小写英文、下划线分隔：

```text
user
developer
moderator
admin
super_admin
system_maintenance
```

scope ID 使用小写英文、下划线分隔：

```text
self
owned
bound_profile
public
assigned
any
system
```

数据库表名使用复数或明确实体名，避免抽象缩写。

## 17. 模型边界

权限模型只回答：

```text
这个 actor 是否可以在当前入口下对这个资源执行这个操作？
```

它不负责：

- 参数校验。
- 业务状态流转。
- 数据库事务封装。
- UI 展示逻辑。

这些能力应由对应 service 或其他模块处理。

## 18. 结论

站点权限系统应围绕 PermissionSubject 和 PermissionActor 建立。

PermissionSubject 表示用户长期拥有的能力。PermissionActor 表示本次请求在具体会话入口下实际能使用的能力。角色只是权限模板，最终判断只看 actor 的有效权限集合和 scope resolver 结果。

`self/any` 不足以支撑站点长期演进。第一版就应建立 scope 注册表，并提供 `owned`、`bound_profile`、`public`、`assigned`、`system` 等明确范围。Yggdrasil 通过 `bound_profile` 和 session policy 纳入模型；Microsoft 登录通过 `microsoft_state` 和 `owned/self` 权限纳入模型。超级管理员不是权限检查的例外，而是拥有全部权限的主体，并且在受限会话入口下仍应被入口策略收窄。
