# Element Skin 权限模型开发者规范

> 状态：现行规范
>
> 目标读者：站点前端、后端贡献者、OAuth / 第三方应用开发者、启动器与外部工具作者、站点管理员
>
> 参考代码快照：`dev-next`，2026-06-30

## 1. 目标与边界

权限系统是站点所有受保护操作的唯一安全边界。所有需要授权的业务操作必须携带 `PermissionActor`，业务层必须使用 `PermissionActor` 的有效权限集合与 scope resolver 判定操作是否允许。

本文档定义 Element Skin 当前生效的权限模型。它不是草案，也不是实施排期说明。站点 API、OAuth 授权页、管理后台权限编辑、Yggdrasil 会话和 Microsoft 正版角色导入都必须遵循本文档。

规范要求：

- 每个站内用户必须拥有一个 `PermissionSubject`。
- 每个受保护业务操作必须携带一个 `PermissionActor`。
- 权限判断必须位于 service 层或 service 调用的权限模块中。
- 路由层不得直接散落角色判断、管理员判断、超级管理员判断。
- 数据库层不得做权限判断。
- 超级管理员不得绕过权限检查；超级管理员通过 `super_admin` 角色拥有全量权限。
- 站内网页登录、Yggdrasil、Microsoft 正版角色导入必须归一到同一套 `PermissionSubject` / `PermissionActor` 模型。
- 委托访问不得绑定角色；委托访问只能使用逐项授权的 Permission 集合。
- `is_admin`、`is_super_admin`、裸 `user_id` 传递不得作为权限接口。
- 权限数据必须结构化存储，禁止使用 JSON 字符串承载权限、scope、授权范围或策略。

权限模型只回答一个问题：

```text
当前 PermissionActor 是否能在当前会话入口下，对指定资源执行指定操作。
```

权限模型不负责参数校验、业务状态流转、数据库事务封装、页面展示。

### 1.1 第三方开发者契约

第三方应用和外部工具只应依赖以下稳定外部概念：

| 概念 | 外部稳定性 | 说明 |
| --- | --- | --- |
| `permission.code` | 稳定 | OAuth scope、授权页、审计、错误提示和开发者文档使用的权限码 |
| `resource.action.scope` | 稳定 | 权限码格式，三段含义固定 |
| scope 语义 | 稳定 | `self`、`owned`、`bound_profile`、`public`、`assigned`、`any`、`system` 的判定语义固定 |
| `PermissionActor` 行为 | 稳定 | 一次请求只能使用 Actor 中实际存在的权限 |
| 委托授权交集规则 | 稳定 | 第三方应用权限不得超过用户有效权限、应用允许权限、用户授权权限三者交集 |

第三方应用不得依赖以下内部实现细节：

| 概念 | 说明 |
| --- | --- |
| `permission.id` | 结构化整数 ID，仅用于数据库关联、缓存和内部一致性校验 |
| `permission.bit_index` | 运行时 bitset 位序，仅用于服务端权限计算 |
| 内置角色名称 | OAuth 和第三方委托授权不得申请角色，只能申请逐项 Permission |
| 数据库表结构 | 表结构用于解释权限模型事实来源，不作为外部远程调用接口 |

第三方应用申请站点能力时，必须使用 `permission.code` 作为授权项。用户批准后，应用拿到的访问令牌只能调用同一套站点 API；后端会把令牌解析成 `PermissionActor` 并执行同样的权限检查。

## 2. 核心对象

### 2.1 PermissionSubject

`PermissionSubject` 表示用户或系统任务的长期权限身份。

字段定义：

| 字段 | 类型 | 约束 | 说明 |
| --- | --- | --- | --- |
| `id` | TEXT | PRIMARY KEY | 权限主体 ID |
| `kind` | TEXT | `user` / `system` | 主体类型 |
| `user_id` | TEXT | 用户主体唯一，系统主体为空 | 关联用户 |
| `status` | TEXT | `active` / `disabled` / `locked` | 主体状态 |
| `created_at` | BIGINT | NOT NULL | 创建时间 |
| `updated_at` | BIGINT | NOT NULL | 更新时间 |

状态规则：

- `active`：主体可参与权限计算。
- `disabled`：主体有效权限集合为空。
- `locked`：主体有效权限集合为空。

账号封禁不等同于 `PermissionSubject.status`。账号封禁属于账号制裁状态，必须参与会话入口策略。当前封禁语义只禁止 Yggdrasil `join` 与 `hasJoined` 代表的服务器加入链路。

### 2.2 PermissionActor

`PermissionActor` 表示一次请求在特定会话入口下实际可用的权限身份。

字段定义：

| 字段 | 类型 | 约束 | 说明 |
| --- | --- | --- | --- |
| `subject_id` | TEXT | NOT NULL | 权限主体 ID |
| `user_id` | TEXT | 用户 actor 必填 | 归属用户 |
| `session_kind` | TEXT | NOT NULL | 会话种类 |
| `entrypoint` | TEXT | NOT NULL | 入口名称 |
| `session_id` | TEXT | 可为空 | 会话、token、state 或任务 ID |
| `bound_profile_id` | TEXT | 可为空 | Yggdrasil 绑定角色 |
| `delegation_id` | TEXT | 可为空 | 委托授权 ID |
| `delegated_client_id` | TEXT | 可为空 | 委托客户端 ID |
| `permissions` | PermissionBitSet | NOT NULL | 本次请求可用权限位图 |

`PermissionActor.permissions` 的计算公式固定为：

```text
subject_effective_bitset
& session_permission_bitset(session_kind, entrypoint)
& delegation_permission_bitset(delegation_id)
&^ account_sanction_denied_bitset(user_id, session_kind, entrypoint)
```

`delegation_id IS NULL` 时，`delegation_permission_policy` 返回全集。`delegation_id IS NOT NULL` 时，`delegation_permission_policy` 只能返回用户逐项授予且客户端被允许申请的权限集合。

### 2.3 Permission

`Permission` 是最小授权单元。

权限拥有两个稳定标识：

| 标识 | 用途 |
| --- | --- |
| `id` | 数据库关联、缓存、结构化权限编码 |
| `code` | 人类可读、审计、管理界面、种子数据 |

`code` 命名格式固定为严格三段：

```text
resource.action.scope
```

`resource`、`action`、`scope` 字段必须结构化存储。系统不得通过 split `code` 解析结构。

`resource` 使用小写英文和下划线，不得包含点号。

字段定义：

| 字段 | 类型 | 约束 | 说明 |
| --- | --- | --- | --- |
| `id` | BIGINT | PRIMARY KEY | 结构化权限整数 ID |
| `code` | TEXT | UNIQUE | 人类可读稳定权限码 |
| `bit_index` | INTEGER | UNIQUE | 运行时稠密 bitset 位序 |
| `resource_id` | INTEGER | NOT NULL | 资源编号 |
| `action_id` | INTEGER | NOT NULL | 动作编号 |
| `scope_id` | INTEGER | NOT NULL | 作用范围编号 |
| `description` | TEXT | NOT NULL | 权限说明 |
| `created_at` | BIGINT | NOT NULL | 创建时间 |

`code` 必须等于 `resource + "." + action + "." + scope_id`。

`id` 使用 64-bit 结构化整数编码并存入数据库 `BIGINT`。数据库使用 PostgreSQL signed `BIGINT`，因此最高位不得使用，保证所有权限 ID 都落在正数范围内稳定存储。

编码布局固定为：

```text
reserved/category resource action scope
16 bits           16 bits  16 bits 16 bits
```

编码公式固定为：

```text
permission_id =
  resource_id << 32 |
  action_id   << 16 |
  scope_id
```

当前版本不启用 `reserved/category` 段，该段必须为 `0`。保留段仍然是物理布局的一部分，权限 ID 的完整布局始终是 `16 * 4`。

约束：

```text
0 < resource_id <= 65535
0 < action_id <= 65535
0 < scope_id <= 65535
reserved/category == 0
```

`resource_id`、`action_id`、`scope_id` 必须由种子数据固定分配。不得使用数据库自增 ID 作为 permission id。

`id` 与 `code` 必须互相校验：

```text
id.resource_id == permission_resources.id
id.action_id == permission_actions.id
id.scope_id == permission_scopes.id
code == resource.code + "." + action.code + "." + scope.code
```

运行时不得用字符串集合做权限计算。权限集合必须编译为 bitset：

```text
permission_bitset[permission.bit_index] = 1
```

`bit_index` 是稠密连续位序，只服务运行时 bitset。`bit_index` 不得等于 `id`，不得参与审计和外部接口。

权限判断固定为整数位运算：

```text
Has(actor.permissions, permission_id)
```

### 2.4 Role

`Role` 是权限主体的权限集合模板。业务层不得判断角色，只能判断权限。

字段定义：

| 字段 | 类型 | 约束 | 说明 |
| --- | --- | --- | --- |
| `id` | TEXT | PRIMARY KEY | 角色 ID |
| `name` | TEXT | UNIQUE | 展示名称 |
| `description` | TEXT | NOT NULL | 角色说明 |
| `system_role` | BOOLEAN | NOT NULL | 系统内置角色 |
| `protected` | BOOLEAN | NOT NULL | 是否为受保护内置角色 |
| `created_at` | BIGINT | NOT NULL | 创建时间 |
| `updated_at` | BIGINT | NOT NULL | 更新时间 |

内置角色不得删除。内置角色的权限变更必须写入审计日志。

同一个 `PermissionSubject` 能绑定多个角色。主体有效权限由所有角色权限与主体覆盖项合并得到。

角色只能绑定到 `PermissionSubject`。委托访问、外部客户端、临时 state、access token 不得绑定角色。

### 2.5 DelegatedPermissionGrant

`DelegatedPermissionGrant` 表示用户把部分权限逐项授予某个客户端。它不是角色，也不得引用角色。

字段定义：

| 字段 | 类型 | 约束 | 说明 |
| --- | --- | --- | --- |
| `id` | TEXT | PRIMARY KEY | 委托授权 ID |
| `user_id` | TEXT | NOT NULL | 授权用户 |
| `subject_id` | TEXT | NOT NULL | 授权用户的权限主体 |
| `client_id` | TEXT | NOT NULL | 客户端 ID |
| `status` | TEXT | `active` / `revoked` | 授权状态 |
| `created_at` | BIGINT | NOT NULL | 创建时间 |
| `revoked_at` | BIGINT | 可为空 | 撤销时间 |

委托访问的最终权限必须满足：

```text
PermissionActor.permissions bitset
<= subject_effective_bitset
PermissionActor.permissions bitset
<= client_allowed_permission_bitset
PermissionActor.permissions bitset
<= delegated_grant_permission_bitset
```

用户权限被 deny 后，委托访问必须在下一次 Actor 构造时失去该权限。

### 2.6 PermissionScope

`PermissionScope` 定义权限范围及 resolver。scope 必须先登记后使用。

首批 scope 固定为：

| scope | resolver_key | 语义 |
| --- | --- | --- |
| `self` | `subject_user` | actor 操作自己的账号级资源 |
| `owned` | `resource_owner` | actor 操作归属于自己的业务资源 |
| `bound_profile` | `session_bound_profile` | actor 操作当前会话绑定的角色 |
| `public` | `resource_public` | actor 读取公开资源 |
| `assigned` | `resource_assignment` | actor 操作明确分配给自己的资源 |
| `any` | `global` | actor 操作任意用户或全站资源 |
| `system` | `system_actor` | 系统主体执行维护任务 |

`self` 和 `any` 不足以覆盖站点权限。`owned`、`bound_profile`、`public`、`assigned`、`system` 必须从第一版进入模型。

## 3. Scope Resolver 规则

### 3.1 `subject_user`

输入：

```text
actor.user_id
resource.user_id
```

通过条件：

```text
actor.user_id == resource.user_id
```

适用资源：

```text
account
```

### 3.2 `resource_owner`

输入：

```text
actor.user_id
resource.owner_user_id
```

通过条件：

```text
actor.user_id == resource.owner_user_id
```

适用资源：

```text
profile
texture
wardrobe
notice
yggdrasil_session
microsoft_import
```

### 3.3 `session_bound_profile`

输入：

```text
actor.user_id
actor.bound_profile_id
profile.id
profile.user_id
```

通过条件：

```text
actor.bound_profile_id IS NOT NULL
AND actor.bound_profile_id == profile.id
AND actor.user_id == profile.user_id
```

适用资源：

```text
profile
texture application target
yggdrasil_server join target
```

### 3.4 `resource_public`

输入：

```text
resource.public
resource.enabled
```

通过条件：

```text
resource.public == true
AND resource.enabled == true
```

适用资源：

```text
texture
profile lookup result
site public setting
```

### 3.5 `resource_assignment`

输入：

```text
actor.subject_id
assignment.subject_id
assignment.resource_id
```

通过条件：

```text
assignment.subject_id == actor.subject_id
AND assignment.resource_id == resource.id
AND assignment.status == 'active'
```

适用资源：

```text
moderation task
review queue item
```

### 3.6 `global`

输入：

```text
actor.permissions
```

通过条件：

```text
actor.permissions contains requested permission
```

`global` 不检查资源归属。

### 3.7 `system_actor`

输入：

```text
actor.subject.kind
actor.session_kind
```

通过条件：

```text
actor.subject.kind == 'system'
AND actor.session_kind == 'system'
```

普通用户和管理员不得使用 `system` scope。

## 4. Permission 目录

### 4.0 粒度规则

Permission 必须对应一个可独立授权、可独立撤销、可独立审计的业务能力。

禁止把安全语义不同的字段合并到同一个 `update` 权限中。一个操作同时修改多个安全语义时，service 必须分别检查对应 Permission。

当前必须拆分的高风险能力：

```text
材质备注、材质模型等普通元数据修改：texture.update_metadata.*
材质公开状态修改：texture.update_visibility.*
角色基础资料修改：profile.update.*
角色当前材质修改：texture.apply.* / texture.clear.*
账号展示资料修改：account.update.*
账号密码修改：account_password.update.*
```

委托访问授权页面必须展示 Permission 粒度的授权项，不得展示角色。


## 7. Yggdrasil 权限流程

### 7.1 authenticate

流程固定为：

```text
1. 校验 username/password。
2. 定位 user。
3. 定位 PermissionSubject。
4. 构造 session_kind=yggdrasil、entrypoint=authserver 的 PermissionActor。
5. 检查 subject.status == active。
6. 检查 yggdrasil_session.create.owned。
7. 查询可用 profile。
8. 按现有 Yggdrasil 规则确定 selectedProfile。
9. 如果 selectedProfile 存在，检查 profile.read.owned。
10. 创建 access token。
```

失败规则：

- 凭据错误返回 Yggdrasil `ForbiddenOperationException`。
- 主体不存在、disabled、locked 返回 Yggdrasil `ForbiddenOperationException`。
- 权限不足返回 Yggdrasil `ForbiddenOperationException`。
- 任一步失败不得创建 token。

### 7.2 refresh

流程固定为：

```text
1. 读取 access token。
2. 校验 clientToken。
3. 根据 token.user_id 定位 PermissionSubject。
4. 构造 session_kind=yggdrasil、entrypoint=authserver 的 PermissionActor。
5. 检查 subject.status == active。
6. 检查 yggdrasil_session.refresh.owned。
7. 如果 token 已绑定 profile，检查 profile 仍属于 token.user_id。
8. 如果请求选择 profile，检查 token 当前未绑定 profile。
9. 检查被选择 profile 属于 token.user_id。
10. 轮换 access token。
```

失败规则：

- 权限、主体状态失败不得产生新 token。
- profile 所属关系失败必须使 refresh 失败。
- refresh 成功后旧 token 必须失效。

### 7.3 validate

流程固定为：

```text
1. 读取 access token。
2. 校验 clientToken。
3. 检查 token 未过期。
4. 根据 token.user_id 定位 PermissionSubject。
5. 构造 session_kind=yggdrasil、entrypoint=authserver 的 PermissionActor。
6. 检查 subject.status == active。
7. 检查 yggdrasil_session.validate.owned。
8. 如果 token 已绑定 profile，检查 profile 仍属于 token.user_id。
```

### 7.4 invalidate / signout

`invalidate` 流程固定为：

```text
1. 读取 access token。
2. 构造 session_kind=yggdrasil、entrypoint=authserver 的 PermissionActor。
3. 检查 yggdrasil_session.invalidate.owned。
4. 删除指定 token。
```

`signout` 流程固定为：

```text
1. 校验 username/password。
2. 定位 PermissionSubject。
3. 构造 session_kind=yggdrasil、entrypoint=authserver 的 PermissionActor。
4. 检查 yggdrasil_session.signout.owned。
5. 删除该用户全部 Yggdrasil token。
```

### 7.5 join

`join` 是服务器登录授权操作，必须进行权限检查。

流程固定为：

```text
1. 读取 access token。
2. token.ProfileID 必须存在。
3. token.ProfileID 必须等于请求 selectedProfile。
4. 根据 token.user_id 定位 PermissionSubject。
5. 构造 session_kind=yggdrasil、entrypoint=sessionserver_join 的 PermissionActor。
6. actor.bound_profile_id = token.ProfileID。
7. 检查 subject.status == active。
8. PermissionActor 构造阶段根据 users.banned_until 移除 yggdrasil_server.join.bound_profile。
9. 检查 profile 仍属于 token.user_id。
10. 执行 RequireScoped(actor, yggdrasil_server.join.bound_profile, profile)。
11. 写入 join session。
```

失败规则：

- token 缺失、未绑定 profile、profile 不匹配、profile 不再属于用户，全部返回 Yggdrasil `Invalid token`。
- subject disabled、locked、权限不足、账号封禁导致权限被移除，全部不得写入 join session。
- 账号封禁返回 Yggdrasil `ForbiddenOperationException`，消息固定为账号被封禁。

### 7.6 hasJoined

`hasJoined` 是 join 授权结果查询，不是公开绕过口。它必须复原 join 时的 actor，并通过 actor permissions 重新确认服务器加入授权仍然有效。

流程固定为：

```text
1. 通过 serverID 读取 join session。
2. join session 不存在或过期，返回 204。
3. 通过 join session.access_token 读取 access token。
4. token 不存在，返回 204。
5. token.ProfileID 必须存在。
6. 读取 profile。
7. profile 必须存在。
8. profile.id 必须等于 token.ProfileID。
9. profile.user_id 必须等于 token.user_id。
10. profile.name 必须等于请求 username。
11. 根据 token.user_id 定位 PermissionSubject。
12. 构造 session_kind=yggdrasil_join_grant、entrypoint=sessionserver_hasjoined 的 PermissionActor。
13. actor.bound_profile_id = token.ProfileID。
14. 检查 subject.status == active。
15. PermissionActor 构造阶段根据 users.banned_until 移除 yggdrasil_server.hasjoined.bound_profile。
16. 执行 RequireScoped(actor, yggdrasil_server.hasjoined.bound_profile, profile)。
17. 返回已签名的 profile payload。
```

失败规则：

- join session、token、profile、username 任一不匹配，返回 204。
- 账号封禁返回 Yggdrasil `ForbiddenOperationException`，消息固定为账号被封禁。
- subject disabled、locked 或权限不足返回 Yggdrasil `ForbiddenOperationException`。

### 7.7 Yggdrasil 材质接口

上传材质流程固定为：

```text
1. 读取 Bearer token。
2. token.ProfileID 必须等于 URL uuid。
3. 读取 profile。
4. profile.user_id 必须等于 token.user_id。
5. 构造 session_kind=yggdrasil、entrypoint=texture 的 PermissionActor。
6. actor.bound_profile_id = token.ProfileID。
7. 检查 subject.status == active。
8. 执行 RequireScoped(actor, texture.apply.bound_profile, profile)。
9. 处理上传文件。
10. 写入材质库。
11. 应用到绑定角色。
```

删除材质流程固定为：

```text
1. 读取 Bearer token。
2. token.ProfileID 必须等于 URL uuid。
3. 读取 profile。
4. profile.user_id 必须等于 token.user_id。
5. 构造 session_kind=yggdrasil、entrypoint=texture 的 PermissionActor。
6. actor.bound_profile_id = token.ProfileID。
7. 检查 subject.status == active。
8. 执行 RequireScoped(actor, texture.clear.bound_profile, profile)。
9. 清除绑定角色材质。
```

## 8. Microsoft 正版角色导入权限流程

Microsoft 正版角色导入是站内用户用外部凭据证明角色所有权的流程。该流程不创建新权限主体，不创建长期外部身份关系，不参与站内登录授权。

### 8.1 auth-url

流程固定为：

```text
1. 当前请求必须是 web actor。
2. 检查 subject.status == active。
3. 检查 microsoft_import.start.owned。
4. 创建 microsoft_state。
```

`microsoft_state` 字段固定为：

| 字段 | 说明 |
| --- | --- |
| `token_hash` | state token 哈希 |
| `user_id` | 当前用户 |
| `subject_id` | 当前权限主体 |
| `purpose` | `profile_import` |
| `created_at` | 创建时间 |
| `expires_at` | 过期时间 |
| `consumed_at` | 消费时间 |

### 8.2 callback

流程固定为：

```text
1. 校验 state。
2. state 必须未过期。
3. state 必须未消费。
4. 交换 Microsoft 角色资料。
5. 写入短期 microsoft_state_result。
6. 标记原 state 已消费。
```

callback 不创建完整站内会话，不写入长期外部身份记录。

### 8.3 get-profile

流程固定为：

```text
1. 当前请求必须是 web actor。
2. 读取 microsoft_state_result。
3. result.user_id 必须等于 actor.user_id。
4. result.subject_id 必须等于 actor.subject_id。
5. result 未过期且未消费。
6. 检查 microsoft_import.read_profile.owned。
7. 返回 Microsoft profile 列表。
```

### 8.4 import-profile

流程固定为：

```text
1. 当前请求必须是 web actor。
2. 读取 microsoft_state_result。
3. result.user_id 必须等于 actor.user_id。
4. result.subject_id 必须等于 actor.subject_id。
5. result 未过期且未消费。
6. 检查 microsoft_import.create_profile.owned。
7. 检查 profile.create.owned。
```

## 9. 账号封禁模型

账号封禁使用结构化字段：

```text
users.banned_until
```

封禁语义：

- `banned_until IS NULL`：未封禁。
- `banned_until <= now`：封禁已过期。
- `banned_until > now`：封禁生效。

封禁生效时必须拒绝：

```text
yggdrasil_server.join.bound_profile
yggdrasil_server.hasjoined.bound_profile
```

封禁不移除：

```text
yggdrasil_session.create.owned
yggdrasil_session.refresh.owned
yggdrasil_session.validate.owned
yggdrasil_session.invalidate.owned
yggdrasil_session.signout.owned
texture.apply.bound_profile
texture.clear.bound_profile
```

封禁生效时不得自动删除：

```text
PermissionSubject
subject_roles
subject_permission_overrides
```

封禁与主体状态的区别：

| 状态 | 影响 |
| --- | --- |
| `banned_until > now` | 禁止加入服务器，不禁止 Yggdrasil 会话创建、刷新、校验、退出或材质写入 |
| `PermissionSubject.status = disabled` | 所有权限为空 |
| `PermissionSubject.status = locked` | 所有权限为空 |

管理员封禁操作必须检查：

```text
account.ban.any
```

管理员解封操作必须检查：

```text
account.unban.any
```

普通管理员不得封禁、解封、修改受保护主体所属账号。该限制不是权限短路，而是业务规则：目标主体拥有受保护角色时，操作 actor 必须具备 `permission_protected.manage.any`。该权限只绑定给受保护管理者角色。

## 10. 权限判断接口

### 10.1 Require

输入：

```text
actor
permission_id
```

规则：

```text
actor.permissions contains permission_id
```

### 10.2 RequireScoped

输入：

```text
actor
permission_id
resource
```

规则：

```text
actor.permissions contains permission_id
AND resolver(permission.scope_id, actor, resource) == true
```

### 10.3 RequireAnyScoped

输入：

```text
actor
permission_ids[]
resource
```

规则：

```text
存在一个 permission_id：
  actor.permissions contains permission_id
  AND resolver(permission.scope_id, actor, resource) == true
```

### 10.4 Forbidden 语义

站内 API：

```text
未认证 -> 401
权限不足 -> 403
私有资源不可见 -> 404
```

Yggdrasil API：

```text
无效 token / profile mismatch -> ForbiddenOperationException: Invalid token.
服务器加入链路账号封禁 -> ForbiddenOperationException: Account is banned. Please contact administrator.
权限不足 -> ForbiddenOperationException
```

## 12. 权限计算

### 12.1 Subject Effective Permissions

计算公式：

```text
role_allow_bitset
| subject_allow_bitset
&^ subject_deny_bitset
```

计算步骤：

```text
1. 读取 permission_subjects。
2. status != active 时返回空集合。
3. 读取 subject_roles。
4. 读取 role_permissions。
5. 读取 subject_permission_overrides。
6. 按 permissions.bit_index 合并 role allow bitset。
7. 按 permissions.bit_index 合并 subject allow bitset。
8. 按 permissions.bit_index 移除 subject deny bitset。
9. 返回 subject effective bitset。
```

### 12.2 Actor Permissions

计算公式：

```text
subject_effective_bitset
& session_permission_bitset(session_kind, entrypoint)
& delegation_permission_bitset(delegation_id)
&^ account_sanction_denied_bitset(user_id, session_kind, entrypoint)
```

委托权限策略：

```text
delegation_id IS NULL：
  delegation_permission_bitset = all permission bits

delegation_id IS NOT NULL：
  delegated_permission_grants.status == active
  delegated_clients.status == active
  delegated_permission_grants.user_id == actor.user_id
  delegated_permission_grants.subject_id == actor.subject_id
  delegated_permission_grants.client_id == actor.delegated_client_id
  delegation_permission_bitset =
    delegated_client_permission_bitset(client_id)
    & delegated_grant_permission_bitset(grant_id)
```

委托访问不得读取 role_permissions，不得通过角色扩大权限。用户撤销授权、客户端禁用、用户权限 deny、角色权限变化都必须影响下一次 Actor 构造结果。

超级管理员通过 `super_admin` 角色获得 subject effective bitset 中的全量非 system 权限。进入 Yggdrasil、Microsoft state 等受限入口时，仍然必须被 session policy 收窄。

## 13. 权限缓存

缓存键固定为：

```text
permission:subject:{subject_id}:effective
permission:actor:{session_kind}:{entrypoint}:{session_id}
permission:delegation:{grant_id}:effective
permission:user:{user_id}:subject
```

必须失效的事件：

```text
permission_subjects.status 变化
subject_roles 变化
role_permissions 变化
subject_permission_overrides 变化
permissions 变化
permission_scopes 变化
session_permission_policies 变化
delegated_clients.status 变化
delegated_client_permissions 变化
delegated_permission_grants.status 变化
delegated_grant_permissions 变化
users.banned_until 变化
用户密码修改
用户删除
Yggdrasil token 创建、刷新、删除
Yggdrasil token 绑定角色变化
Microsoft state 创建、消费、过期
```

缓存不是事实来源。数据库是唯一事实来源。

## 14. Service 层规范

受保护 service 方法签名必须包含 `PermissionActor`。

固定形式：

```go
func Operation(ctx context.Context, actor PermissionActor, input Input) (Output, error)
```

service 层流程固定为：

```text
1. 读取业务资源。
2. 资源不存在时返回业务定义的 not found。
3. 调用 Require / RequireScoped / RequireAnyScoped。
4. 权限通过后执行状态校验。
5. 执行数据库写入。
```

禁止形式：

```go
func Operation(ctx context.Context, userID string, isAdmin bool, input Input) (Output, error)
```

## 16. 开发者准则

权限系统由五个层次组成：

```text
PermissionSubject：长期权限身份
Subject Effective Permissions：主体长期有效权限
PermissionActor：会话入口下的实际权限身份
Delegation Permission Policy：委托访问的逐项权限收窄
Scope Resolver：资源级权限判定
```

`self` 和 `any` 不能覆盖站点需求。权限范围固定包含 `owned`、`bound_profile`、`public`、`assigned`、`system`。Yggdrasil 服务器登录 `join` 与 `hasJoined` 是权限控制的一部分。账号封禁通过会话入口策略移除服务器加入链路权限。Microsoft 正版角色导入使用临时 state 入口收窄权限，不产生长期外部身份关系，不产生新权限主体。委托访问使用逐项 Permission 授权，不得使用角色扩大权限。
