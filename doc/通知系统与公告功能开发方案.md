# 通知系统与公告功能开发方案

## 1. 背景

Element Skin 后续计划提供 OAuth 开发者应用、第三方应用授权、可能的外置插件系统等能力。站点需要一个统一的信息触达入口，让管理员可以向用户发布站内公告，也让系统后续可以向指定用户投递 OAuth 审核、授权变更、安全提醒等个人通知。

因此本方案不单独设计一个临时的公告表，而是设计一套统一的 Notice 通知系统。公告是通知系统中的一种通知类型。

## 2. 目标

第一阶段实现以下能力：

- 管理员可以创建、编辑、删除公告。
- 公告支持短公告和长公告。
- 公告正文支持受限 Markdown。
- 仪表盘改为左右两列布局，左侧保留现有资源和服务状态，右侧展示公告条目。
- 长公告在仪表盘只显示摘要，点击进入通知详情页查看全文。
- 短公告在仪表盘直接展示正文简略内容，点击同样可以进入详情页。
- 登录用户可以查看通知中心和通知详情。
- 登录用户可以标记通知已读，或关闭可关闭的公告。
- 系统可以向指定用户投递个人通知。
- 系统可以自动清理过期通知。
- 通知可以设置无期限有效。

非目标：

- 第一阶段不支持游客通知。当前除首页外均要求认证，公告入口只面向登录后的站内页面。
- 第一阶段不设计 `super_admins` audience。超级管理员只有一个，不需要作为公告投放人群单独建模。
- 第一阶段不实现邮件、短信、WebSocket 实时推送。
- 第一阶段不实现复杂用户分群，例如按注册时间、角色数量、材质数量投放。
- 第一阶段不把公告展示到首页。首页仍保持当前主视觉和登录入口。

## 3. 核心概念

### 3.1 Notice

Notice 是统一通知实体，可以承载公告、系统通知、OAuth 相关通知等。

第一阶段实际启用：

- `announcement`: 管理员发布的站内公告。
- `system`: 系统级用户事件，例如 OAuth 应用审核结果。

后续可扩展：

- `oauth_app`: OAuth 应用审核、密钥轮换、安全提醒。
- `oauth_grant`: 第三方应用请求新权限、授权被撤销。

### 3.2 短公告与长公告

公告通过 `display_mode` 区分展示方式。

- `inline`: 短公告。仪表盘可以直接显示正文的简略渲染内容。
- `detail`: 长公告。仪表盘只显示 `summary`，点击进入详情页查看完整 Markdown。

无论短公告还是长公告，都可以进入通知详情页。这样公告和通知中心保持统一交互。

### 3.3 Receipt

Receipt 是用户与通知之间的状态记录。

它不需要在公告创建时为所有用户预生成。用户阅读或关闭通知时再写入。

### 3.4 Target

Target 是通知与指定用户之间的投递关系。

它只用于 `audience = targeted` 的通知。公告面向所有用户或管理员时不写 target，避免为大范围通知预生成大量用户行。

## 4. 数据模型

### 4.1 notices

```sql
CREATE TABLE IF NOT EXISTS notices (
    id TEXT PRIMARY KEY,
    type TEXT NOT NULL,
    title TEXT NOT NULL,
    summary TEXT NOT NULL DEFAULT '',
    content_markdown TEXT NOT NULL,
    display_mode TEXT NOT NULL,
    level TEXT NOT NULL DEFAULT 'info',
    link_text TEXT NOT NULL DEFAULT '',
    link_url TEXT NOT NULL DEFAULT '',
    audience TEXT NOT NULL DEFAULT 'users',
    enabled BOOLEAN NOT NULL DEFAULT TRUE,
    pinned BOOLEAN NOT NULL DEFAULT FALSE,
    dismissible BOOLEAN NOT NULL DEFAULT TRUE,
    starts_at BIGINT DEFAULT NULL,
    ends_at BIGINT DEFAULT NULL,
    created_by TEXT,
    created_at BIGINT NOT NULL,
    updated_at BIGINT NOT NULL,
    FOREIGN KEY(created_by) REFERENCES users(id)
);
```

字段说明：

| 字段 | 说明 |
| --- | --- |
| `id` | 通知 ID，使用无横线 UUID。 |
| `type` | 通知类型，第一阶段主要为 `announcement`。 |
| `title` | 标题，列表、仪表盘、详情页均展示。 |
| `summary` | 长公告摘要。`display_mode = detail` 时必填。 |
| `content_markdown` | Markdown 原文。后端只存原文，不存 HTML。 |
| `display_mode` | `inline` 或 `detail`。 |
| `level` | `info`、`success`、`warning`、`danger`。 |
| `link_text` | 可选操作链接文案。 |
| `link_url` | 可选操作链接地址。 |
| `audience` | 可见人群。允许 `users`、`admins`、`targeted`。 |
| `enabled` | 是否启用。未启用的通知不进入用户侧列表。 |
| `pinned` | 是否置顶。 |
| `dismissible` | 是否允许用户关闭。 |
| `starts_at` | 生效时间，`NULL` 表示立即生效。 |
| `ends_at` | 过期时间，`NULL` 表示无期限限制。 |
| `created_by` | 创建公告的管理员用户 ID。 |
| `created_at` | 创建时间，毫秒时间戳。 |
| `updated_at` | 更新时间，毫秒时间戳。 |

约束建议：

- `type` 第一阶段允许 `announcement`。
- `display_mode` 允许 `inline`、`detail`。
- `level` 允许 `info`、`success`、`warning`、`danger`。
- `audience` 允许 `users`、`admins`、`targeted`。
- `audience = targeted` 时必须写入至少一个目标用户。
- `display_mode = detail` 时 `summary` 不允许为空。
- `summary` 建议限制 160 字以内。
- `title` 建议限制 80 字以内。
- `content_markdown` 建议限制 20KB 以内。
- `starts_at` 和 `ends_at` 同时存在时，`ends_at` 必须大于 `starts_at`。

索引建议：

```sql
CREATE INDEX IF NOT EXISTS idx_notices_active
ON notices (enabled, audience, pinned, starts_at, ends_at, created_at);

CREATE INDEX IF NOT EXISTS idx_notices_cleanup
ON notices (ends_at)
WHERE ends_at IS NOT NULL;
```

### 4.2 notice_targets

```sql
CREATE TABLE IF NOT EXISTS notice_targets (
    notice_id TEXT NOT NULL,
    user_id TEXT NOT NULL,
    created_at BIGINT NOT NULL,
    PRIMARY KEY(notice_id, user_id),
    FOREIGN KEY(notice_id) REFERENCES notices(id) ON DELETE CASCADE,
    FOREIGN KEY(user_id) REFERENCES users(id) ON DELETE CASCADE
);
```

字段说明：

| 字段 | 说明 |
| --- | --- |
| `notice_id` | 通知 ID。 |
| `user_id` | 目标用户 ID。 |
| `created_at` | 投递关系创建时间。 |

索引建议：

```sql
CREATE INDEX IF NOT EXISTS idx_notice_targets_user
ON notice_targets (user_id, notice_id);
```

Target 行为：

- 只有 `audience = targeted` 的通知允许写入 target。
- `target_user_ids` 必须去重，空字符串必须拒绝或过滤。
- 用户侧列表与详情必须通过 `notice_targets` 判断可见性。
- 删除 notice 时通过 `ON DELETE CASCADE` 清理关联 target。

### 4.3 notice_receipts

```sql
CREATE TABLE IF NOT EXISTS notice_receipts (
    notice_id TEXT NOT NULL,
    user_id TEXT NOT NULL,
    read_at BIGINT DEFAULT NULL,
    dismissed_at BIGINT DEFAULT NULL,
    created_at BIGINT NOT NULL,
    PRIMARY KEY(notice_id, user_id),
    FOREIGN KEY(notice_id) REFERENCES notices(id) ON DELETE CASCADE,
    FOREIGN KEY(user_id) REFERENCES users(id) ON DELETE CASCADE
);
```

字段说明：

| 字段 | 说明 |
| --- | --- |
| `notice_id` | 通知 ID。 |
| `user_id` | 用户 ID。 |
| `read_at` | 用户已读时间。 |
| `dismissed_at` | 用户关闭时间。 |
| `created_at` | receipt 首次创建时间。 |

索引建议：

```sql
CREATE INDEX IF NOT EXISTS idx_notice_receipts_user
ON notice_receipts (user_id, read_at, dismissed_at);
```

Receipt 行为：

- 用户首次读通知详情时 upsert receipt，并设置 `read_at`。
- 用户关闭可关闭公告时 upsert receipt，并设置 `dismissed_at`。
- `dismissed_at` 不为空的公告不再出现在仪表盘公告列表。
- `read_at` 不为空但 `dismissed_at` 为空的公告仍可在通知中心出现，但标记为已读。
- 删除 notice 时通过 `ON DELETE CASCADE` 清理关联 receipt。

## 5. 可见性规则

用户侧只面向已登录用户。

`audience` 规则：

- `users`: 所有登录用户可见，管理员也可见。
- `admins`: 仅管理员可见。
- `targeted`: 仅 `notice_targets` 中存在当前用户的通知可见。

用户侧查询需要同时满足：

- `enabled = true`
- `starts_at IS NULL OR starts_at <= now`
- `ends_at IS NULL OR ends_at > now`
- audience 与当前用户身份匹配
- 若 `audience = targeted`，`notice_targets` 中必须存在当前用户
- 当前用户未关闭，即 receipt 不存在或 `dismissed_at IS NULL`

通知详情页：

- 已过期通知不允许用户侧访问，返回 404。
- 未启用通知不允许用户侧访问，返回 404。
- audience 不匹配时返回 404，避免泄露通知存在性。
- 用户有权限访问详情时，可以标记已读。

管理侧：

- 管理员可以查看所有未删除通知，包括禁用和过期通知。
- 管理员可以删除通知。删除是真删除，避免历史垃圾积累。

## 6. 过期清理策略

通知支持无期限有效：

- `ends_at = NULL` 表示永不过期，不参与自动删除。
- `ends_at` 有值且小于当前时间，表示已过期。

后端启动时和后台定时任务中执行过期清理：

- 建议每小时执行一次。
- 删除 `ends_at IS NOT NULL AND ends_at < now - retention` 的通知。
- `retention` 第一阶段可以为 0，即过期后即可删除。
- 如果担心时间误差，可以设置为 24 小时宽限，但产品上不强制。

由于用户明确允许公告删除，本功能不需要软删除。删除 notice 后 receipt 自动级联删除。

建议在 `app.New` 中新增清理循环，模式参考现有 refresh token cleaner。

### 7.3 service notice

Service 层负责业务校验：

- 创建和更新时校验枚举值。
- 长公告必须填写 summary。
- 短公告可以不写 summary。
- 链接 URL 必须安全。
- 标题、摘要、正文长度限制。
- starts_at、ends_at 时间窗口校验。
- 详情访问校验 audience、enabled、时间窗口。
- dismiss 时校验 notice 是否允许关闭。

## 8. API 设计

### 8.1 用户侧 API

所有用户侧通知 API 都需要登录。

#### 获取通知列表

```text
GET /notices?type=announcement&limit=20&cursor=&include_read=true&dashboard=false
```

响应示例：

```json
{
  "items": [
    {
      "id": "notice-id",
      "type": "announcement",
      "title": "开发者中心即将开放",
      "summary": "OAuth 应用注册入口即将上线，请查看详情了解权限说明。",
      "content_preview": "",
      "display_mode": "detail",
      "level": "info",
      "link_text": "查看详情",
      "link_url": "/notifications/notice-id",
      "pinned": true,
      "dismissible": true,
      "read": false,
      "created_at": 1760000000000,
      "updated_at": 1760000000000
    }
  ],
  "next_cursor": ""
}
```

说明：

- `display_mode = detail` 时，列表返回 `summary`。
- `display_mode = inline` 时，可以返回 `content_preview`，由后端截断 Markdown 原文或前端自行显示正文前若干行。建议第一阶段直接返回 `content_markdown`，由前端控制最大高度和渐隐。
- `dashboard=true` 时默认排除已关闭公告，限制返回公告类型，排序优先置顶。

#### 获取通知详情

```text
GET /notices/{id}
```

响应示例：

```json
{
  "id": "notice-id",
  "type": "announcement",
  "title": "开发者中心即将开放",
  "summary": "OAuth 应用注册入口即将上线，请查看详情了解权限说明。",
  "content_markdown": "完整 Markdown 内容",
  "display_mode": "detail",
  "level": "info",
  "link_text": "前往开发者中心",
  "link_url": "/developer",
  "pinned": true,
  "dismissible": true,
  "read": true,
  "created_at": 1760000000000,
  "updated_at": 1760000000000
}
```

详情读取可以自动标记已读。若希望更明确，也可以前端进入详情后调用 `POST /notices/{id}/read`。

#### 标记已读

```text
POST /notices/{id}/read
```

行为：

- 幂等。
- 通知不存在、无权限、过期、禁用时返回 404。
- 成功返回 204。

#### 关闭通知

```text
POST /notices/{id}/dismiss
```

行为：

- 幂等。
- `dismissible = false` 返回 403。
- 成功返回 204。

### 8.2 管理侧 API

所有管理侧 API 走现有管理员认证。

#### 管理员列表

```text
GET /admin/notices?type=announcement&status=all&limit=20&cursor=
```

`status` 可选：

- `all`
- `enabled`
- `disabled`
- `expired`
- `scheduled`

#### 创建通知

```text
POST /admin/notices
```

请求示例：

```json
{
  "type": "announcement",
  "title": "开发者中心即将开放",
  "summary": "OAuth 应用注册入口即将上线，请查看详情了解权限说明。",
  "content_markdown": "完整 Markdown 内容",
  "display_mode": "detail",
  "level": "info",
  "link_text": "查看详情",
  "link_url": "/notifications/notice-id",
  "audience": "users",
  "target_user_ids": [],
  "enabled": true,
  "pinned": true,
  "dismissible": true,
  "starts_at": null,
  "ends_at": null
}
```

#### 更新通知

```text
PATCH /admin/notices/{id}
```

支持局部更新，但语义是替换发布：删除旧通知并创建新通知。旧通知的已读、忽略、历史内容不迁移到新通知。`audience = targeted` 的通知替换时继续沿用原目标用户。

#### 删除通知

```text
DELETE /admin/notices/{id}
```

行为：

- 真删除。
- 关联 receipt 自动删除。
- 不存在返回 404。
- 成功返回 204。

## 9. Markdown 安全策略

后端只保存 Markdown 原文，不渲染 HTML。

前端渲染建议：

- 使用 `markdown-it`。
- 配置 `html: false`，禁用原始 HTML。
- 渲染后的 HTML 再经过 `DOMPurify`。
- 链接点击默认 `rel="noopener noreferrer"`。

允许链接：

- 站内相对路径，例如 `/developer`、`/notifications/xxx`。
- `http://`
- `https://`

禁止链接：

- `javascript:`
- `data:`
- `vbscript:`
- 空白控制字符伪装协议。

第一阶段允许的 Markdown 能力：

- 标题。
- 段落。
- 加粗、斜体、删除线。
- 行内代码和代码块。
- 有序/无序列表。
- 引用。
- 链接。

第一阶段不建议启用：

- 原始 HTML。
- iframe。
- 任意 style。
- 图片上传。
- 复杂表格。

管理后台需要预览，预览必须与用户侧使用同一套 Markdown 渲染和清洗逻辑。

## 10. 前端页面设计

### 10.1 仪表盘布局

当前仪表盘需要从单列改为响应式左右两列。

桌面端：

```text
┌───────────────────────────────────────────────┬──────────────────────┐
│ 左侧主内容                                      │ 右侧公告列             │
│ - 资源统计卡片                                  │ - 公告条目 1           │
│ - 快速接入启动器                                │ - 公告条目 2           │
│ - 服务状态                                      │ - 公告条目 3           │
└───────────────────────────────────────────────┴──────────────────────┘
```

建议宽度：

- 左侧：自适应，占主要空间。
- 右侧：固定或受限宽度，建议 `320px` 到 `380px`。
- 两列间距使用现有页面栅格间距。

移动端或窄宽度：

```text
┌─────────────────────────┐
│ 公告区域或主内容         │
├─────────────────────────┤
│ 资源统计卡片             │
├─────────────────────────┤
│ 快速接入启动器           │
├─────────────────────────┤
│ 服务状态                 │
└─────────────────────────┘
```

移动端排列建议：

- 保持一列。
- 公告区域放在仪表盘标题下方、资源统计前。
- 公告条目限制最大高度，避免挤压主体内容。

仪表盘公告条目：

- 使用现有自定义卡片组件风格，不直接裸用 `el-card`。
- 每条公告是一个可点击条目。
- 显示 level 图标或轻量状态标识，避免大色块。
- 显示标题。
- 长公告显示 `summary`。
- 短公告显示正文的简略 Markdown 渲染或纯文本摘取。
- 显示时间。
- 可关闭公告显示关闭按钮。
- 点击条目进入 `/notifications/{id}`。

- 生效时间。
- 过期时间。
- 链接文字。
- 链接地址。
- 预览。

校验：

- 长公告必须填写摘要。
- 标题不能为空。
- Markdown 正文不能为空。
- 链接文字和链接地址建议同时填写或同时为空。
- 链接地址必须通过安全校验。

## 11. 缓存策略

第一阶段可以不做复杂缓存，直接走 PostgreSQL。

原因：

- 公告数量低。
- 用户侧查询频率不会高于现有服务状态和材质列表。
- Receipt 会影响每个用户的可见状态，过早缓存会增加复杂度。

后续如需缓存：

- 可缓存公共的 active notices 基础列表。
- 用户 receipt 单独查询后合并。
- 管理员修改或删除 notice 时清理 notice 缓存。

## 12. 错误处理

用户侧：

- 未登录：401。
- 无权限、禁用、过期、不存在：404。
- 不可关闭却调用 dismiss：403。
- 请求参数非法：400。

管理侧：

- 非管理员：403。
- 数据不存在：404。
- 枚举、时间、长度、链接校验失败：400。

错误格式沿用当前后端 `util.HTTPError` 和 `util.Error`。

## 15. 与 OAuth 功能的衔接

OAuth 功能复用 Notice 系统发布：

- 开发者中心开放公告。
- OAuth 应用审核结果通知。
- 应用请求新权限通知。
- 用户授权被撤销通知。
- 管理员重置应用密钥通知。
- 外置插件系统相关安全提示。

当前 OAuth 应用审核链路的通知规则：

- 开发者创建应用或重新提交审核时，系统创建 `type = system`、`audience = admins` 的待审核通知。
- 管理员通过、驳回或停用应用时，系统创建 `type = system`、`audience = targeted` 的结果通知，并通过 `notice_targets` 投递给应用 owner。
- 驳回和停用必须填写原因，原因写入通知正文。
- OAuth 审核相关通知 `ends_at` 固定为发布后 30 天。

OAuth 个人通知使用同一张 `notices` 表和同一套用户侧通知中心。`notice_targets` 只表达投递目标，`notice_receipts` 只表达用户已读和忽略状态，二者职责不能混用。

## 16. 待确认问题

- 仪表盘公告列在桌面端的最终宽度：建议 340px。
- 移动端公告区域是否放在资源统计之前：建议放在标题下、资源统计前。
- 详情页读取时是否自动标记已读：建议自动标记。
- 过期后是否立即删除：建议立即删除，`ends_at = NULL` 表示不自动删除。
- 第一阶段通知中心需要未读角标。角标统计公告、系统消息以及未来新增通知类型的未读数量。
